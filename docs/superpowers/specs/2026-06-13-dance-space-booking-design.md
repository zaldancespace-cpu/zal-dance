# Сайт танцевального зала «ЗАЛ»: бронирование, оплата, админка

Дата: 2026-06-13
Статус: утвержденный дизайн перед планом внедрения

## Цель

Перевести текущий Vite/React-сайт танцевального зала в рабочую продакшен-схему: убрать остатки фотостудии, подключить новый Firebase, добавить оплату через ЮKassa, email-уведомления, обязательные согласия и админку для управления бронированиями.

## Основные решения

- Рабочая директория для изменений: `/Users/matvejvoroncov/Downloads/lookbeta1/dance-space-zal2`.
- Публичный сайт остается Vite + React + TypeScript + Tailwind + HeroUI.
- Backend добавляется отдельной Node/Express-службой в `server/`, рассчитанной на будущий VPS.
- Секреты ЮKassa, SMTP и Firebase Admin хранятся только на сервере через переменные окружения.
- Бронь становится подтвержденной только после успешной оплаты, подтвержденной сервером через webhook ЮKassa.
- Личный кабинет клиента не нужен. «Ваше бронирование» определяется локально на устройстве через сохраненный booking/payment id и ссылку из письма.
- Админка `/admin` защищается Firebase Auth: email + пароль администратора.

## Данные Firebase

Новый Firebase-проект используется для:

### `bookings`

Документ бронирования:

- `id`
- `date` в формате `YYYY-MM-DD`
- `timeSlots`: массив строк, например `["10:00", "11:00"]`
- `bookingType`: `individual` или `group`
- `customerName`
- `customerEmail`
- `customerPhone`
- `customerNotes`
- `totalPrice`
- `status`: `pending_payment`, `confirmed`, `cancelled`, `expired`
- `paymentId`
- `paymentStatus`
- `createdAt`
- `confirmedAt`
- `cancelledAt`
- `cancelReason`
- `source`: `public`, `admin_manual`

### `blockedSlots`

Ручные блокировки администратора:

- `id`
- `date`
- `timeSlots`
- `reason`
- `createdBy`
- `createdAt`
- `active`

### `paymentEvents`

Журнал webhook-событий ЮKassa:

- `id`
- `paymentId`
- `event`
- `status`
- `payload`
- `createdAt`

### `adminUsers`

Минимальная роль администратора:

- `uid`
- `email`
- `role`: `admin`
- `createdAt`

## Публичное бронирование

1. Клиент выбирает дату, тип занятия и один или несколько слотов.
2. Календарь показывает занятые слоты из `bookings.status == confirmed` и активных `blockedSlots`.
3. Клиент заполняет имя, email, телефон, комментарий.
4. Клиент обязательно принимает чекбоксы согласий.
5. Frontend отправляет запрос на backend `POST /api/create-payment`.
6. Backend повторно проверяет доступность слотов в Firebase.
7. Backend создает `pending_payment` бронь и платеж в ЮKassa.
8. Клиент перенаправляется на страницу оплаты ЮKassa.
9. После webhook со статусом успешной оплаты backend переводит бронь в `confirmed`.
10. Backend отправляет письма клиенту и администратору.
11. Frontend на странице возврата показывает результат оплаты и сохраняет id брони локально для пометки «Ваше бронирование».

## Недоступность времени

Слот считается занятым, если:

- есть `confirmed` бронь на эту дату и время;
- есть активная ручная блокировка на эту дату и время;
- идет создание оплаты на этот слот, и backend временно защищает слот от гонок.

Если два клиента одновременно пытаются оплатить один слот, backend должен не позволить создать два подтвержденных бронирования. На первом этапе защита реализуется повторной серверной проверкой перед созданием платежа и перед подтверждением webhook. При необходимости позже добавляется Firestore transaction/lock-документ на слот.

## Согласия и юридические чекбоксы

В форме бронирования нужны обязательные чекбоксы:

- согласие на обработку персональных данных;
- согласие с правилами аренды зала;
- согласие с условиями оплаты, отмены и возврата;
- подтверждение корректности email/телефона.

Кнопка оплаты неактивна, пока обязательные согласия не приняты.

Нужны отдельные страницы или секции:

- `/privacy` — политика обработки персональных данных;
- `/terms` — правила аренды, оплаты, отмены и возврата.

Тексты можно начать с базовой редакции и затем заменить на юридически проверенные.

## ЮKassa

Интеграция строится только через backend:

- `POST /api/create-payment` создает платеж в ЮKassa и возвращает `confirmation_url`.
- `POST /api/yookassa-webhook` принимает уведомления и подтверждает бронь.
- Секретный ключ ЮKassa не попадает в браузер.
- Idempotence key используется при создании платежа.
- Сумма платежа рассчитывается на сервере, а не доверяется клиенту.
- Webhook проверяет платеж и связывает его с `bookingId`.

## Email

Email отправляется сервером через Nodemailer/SMTP.

Переменные окружения:

- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_SECURE`
- `SMTP_USER`
- `SMTP_PASS`
- `MAIL_FROM`
- `ADMIN_EMAIL`

Для старта можно использовать новую Gmail-почту с App Password. Для продакшена предпочтительна доменная почта.

Письмо клиенту содержит:

- номер брони;
- дату и время;
- тип занятия;
- сумму;
- адрес;
- правила/ссылку на условия;
- контакт для связи;
- ссылку на страницу проверки брони.

Письмо администратору содержит:

- номер брони;
- данные клиента;
- дату и слоты;
- сумму и paymentId;
- комментарий клиента;
- ссылку на `/admin`.

## Админка `/admin`

Админка защищена Firebase Auth email/password.

Функции первой версии:

- вход администратора;
- список броней с фильтром по дате и статусу;
- просмотр деталей брони;
- отмена брони администратором;
- ручная блокировка слотов;
- снятие ручной блокировки;
- отображение статуса оплаты;
- быстрый переход к календарному дню.

Админские операции должны выполняться через backend или через Firestore rules, которые разрешают доступ только админам. Для первого VPS-варианта предпочтительно выполнять критичные операции через backend с проверкой Firebase ID token.

## Backend на VPS

В проект добавляется `server/`:

- Express API;
- Firebase Admin SDK;
- ЮKassa client/fetch layer;
- Nodemailer mailer;
- env validation;
- CORS для домена сайта;
- structured logging;
- health endpoint `/api/health`.

Планируемые endpoint’ы:

- `GET /api/health`
- `POST /api/create-payment`
- `POST /api/yookassa-webhook`
- `GET /api/payment-status/:bookingId`
- `POST /api/admin/cancel-booking`
- `POST /api/admin/block-slots`
- `POST /api/admin/unblock-slots`

При деплое на VPS нужны Node.js, PM2, Nginx, HTTPS, env-файл и публичный webhook URL для ЮKassa.

## Ошибки и edge cases

- Если платеж не завершен, бронь остается `pending_payment` и не занимает слот навсегда.
- `pending_payment` брони старше заданного TTL переводятся в `expired`.
- Если email не отправился после оплаты, бронь остается confirmed, а ошибка логируется для повторной отправки.
- Если webhook пришел повторно, обработка должна быть идемпотентной.
- Если клиент вернулся с оплаты раньше webhook, frontend показывает «проверяем оплату» и опрашивает `/api/payment-status/:bookingId`.
- Если админ отменяет бронь, слот освобождается, а статус становится `cancelled`.

## Внешние источники

- ЮKassa: создание платежа, confirmation_url, webhook и серверная обработка оплаты описаны в официальной документации ЮKassa.

- Nodemailer: SMTP-транспорт используется для серверной отправки email.
